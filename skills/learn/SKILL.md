# learn: интерактивный тьютор AI Architect Curriculum

Тьютор поверх учебной секции `wiki/learning/`. Контентная модель: `AI Architect Curriculum`
(программа + таблица прогресса), `CCA-F Exam` (blueprint), по модулю: глава
`<ID> <Title>.md` + банк вопросов `<ID> Quiz.md`.

**Portability rule:** никаких захардкоженных абсолютных путей. Learning-директория
резолвится так: если в CWD есть `wiki/learning/`, использовать её; иначе `Glob **/learning/AI Architect Curriculum.md`
и взять родительскую папку; если не нашлось, спросить пользователя. Секция спроектирована
под вынос в отдельный репо, skill обязан работать и там.

---

## Mode dispatch

| Аргументы | Режим |
|---|---|
| (пусто) или `progress` | Progress |
| `study <ID>` | Study |
| `quiz <ID>` | Quiz |
| `practice <ID>` | Practice |
| только `<ID>` | посмотреть прогресс модуля: если глава не пройдена, предложить study; если study пройден, предложить quiz |

ID модулей: A1..A5 (exam-core, веса из blueprint) и B1..B7 (foundation). Сверять с
таблицами в `AI Architect Curriculum`.

---

## Progress

1. Прочитать `AI Architect Curriculum`: таблица «Прогресс» + секция «Слабые темы» (если есть).
2. Показать компактно: завершённые / в работе / не начатые, последние quiz-скоры, слабые темы.
3. Рекомендовать следующий шаг по «Рекомендуемому порядку» из curriculum, с учётом весов экзамена.

Read-only режим, pre-flight не нужен.

---

## Study

1. Прочитать главу модуля. Если главы НЕТ, перейти к Authoring (ниже), это единственный
   write-путь в study.
2. Выдать пользователю:
   - конспект ключевых идей (5-8 буллетов, не пересказ всей главы);
   - какие субдомены blueprint покрывает (из `CCA-F Exam`);
   - внешние материалы к прочтению (из `Anthropic Academy Catalog`, колонка «Модули»);
   - оценку времени и ссылку на главу (пусть читает её в Obsidian целиком).
3. Отметить в таблице прогресса: `Глава: прочитана <YYYY-MM-DD>` (после подтверждения
   пользователя, что прочитал; иначе `выдана`).
4. Предложить quiz следующим шагом.

### Authoring (когда главы нет)

Pre-flight: подтвердить у пользователя scope (один модуль) и что пишем сейчас (research
жжёт токены). Затем:

1. **Scope из blueprint**, не из вопросов: субдомены модуля из `CCA-F Exam`.
2. **Research docs-first**: официальные доки и спеки (для Claude API / Claude Code фактов
   сначала локальный skill `claude-api` и context7, web только для гэпов). Никаких фактов
   «по памяти» без источника.
3. **Cross-check по отчётам сдавших** (web): повторяющиеся темы усиливаем, отсутствующие
   в главе дописываем. Отчёты = сигнал акцентов, НЕ источник истины и не учебник.
4. **Написать главу в учебном стиле** (textbook, не конспект; эталон: A4 после rewrite
   2026-06-10). Проза на русском, термины на английском, без U+2014. Обязательная структура:
   - вводная секция «о чём модуль и как читать»: мотивация + roadmap секций + сквозной
     принцип модуля;
   - каждая секция: «какую проблему решаем» → определение термина callout-ом `> [!note]`
     при первом упоминании → пример «плохо / хорошо» (код рядом, не словами) → нюансы →
     «Итоги секции» (3-5 предложений) → «Контрольные вопросы» (2-3 открытых, без ответов);
   - ловушки экзамена врезками `> [!warning] Ловушка экзамена`;
   - повествовательная нить: секции ссылаются друг на друга мостиками, не 7 независимых блоков;
   - в конце: «Экзаменационные анти-паттерны» (таблица symptom / слой поверх / root cause),
     «Шпаргалка для повторения» (плотная выжимка всей главы для прогона перед экзаменом),
     «Practice» (задача + критерии оценки, скрытые до ответа), «Что почитать», «Sources».
   Запрещено: telegraphic-буллеты с 3-4 фактами в строке как основной текст, термины без
   введения (PTC, harness и т.п. определять при первом употреблении). Wikilinks только
   внутри learning/. Frontmatter: type: learning, module, track, weight, status,
   created/updated, tags, related.
5. **Написать банк** `<ID> Quiz.md`: 8-12 вопросов по формату ниже. Вопросы пишем сами,
   в экзаменационном стиле (сценарий production-троблшутинга, правильный ответ чинит root
   cause, дистракторы = «добавить слой поверх» / правдоподобные полумеры). Чужие вопросы
   дословно не копировать.
6. Vault bookkeeping (только если рядом есть `scripts/allocate-address.sh`, т.е. мы в
   вольте, а не в standalone-репо): address через скрипт, sessions: в frontmatter, entry в
   `wiki/log.md`, обновить таблицы curriculum.

## Quiz

1. Прочитать `<ID> Quiz.md`. Содержимое банка пользователю НЕ показывать до ответов.
2. Если в curriculum есть слабые темы по этому модулю, включить их вопросы обязательно.
3. Задавать вопросы через AskUserQuestion, батчами по 2-3 вопроса за вызов. Каждый вопрос:
   header `Q<N>`, текст сценария в question, 4 варианта (формулировки вариантов сократить
   до label ≤ 5 слов + description с полным текстом варианта).
4. Вопросы НЕ переводить на английский, порядок вариантов можно перемешивать (запомнить
   соответствие).
5. После каждого батча: разбор только ошибочных ответов (короткий, из поля «Разбор» банка).
   Верные не пересказывать.
6. Итог: `X/N (YY%)`, аналог проходного = 72% (как 720/1000 на реальном экзамене).
   Темы ошибок собрать по topic-тегам вопросов.
7. Записать в curriculum: колонка Quiz = `X/N YYYY-MM-DD`, ошибочные topic-теги добавить
   в секцию `## Слабые темы` (формат: `- <ID>/<topic> (quiz YYYY-MM-DD)`); тема убирается
   оттуда после квиза, где соответствующий вопрос отвечен верно.

---

## Practice

1. Взять задачу из секции `## Practice` главы. Показать условие, НЕ показывать критерии.
2. Пользователь отвечает в свободной форме (текст, диаграмма словами, JSON).
3. Оценить по критериям из главы: что найдено / что упущено / что лишнее. Формат отзыва:
   сначала вердикт (passed / partial / retry), потом по пунктам.
4. Записать в curriculum: Practice = `passed/partial YYYY-MM-DD`.

## Формат банка вопросов (`<ID> Quiz.md`)

Ответы и разборы НЕ пишутся под вопросом: при чтении банка глазами (в Obsidian, при
самостоятельном прогоне) ответ под вопросом читается раньше, чем человек успел подумать.
Все ответы уходят в секцию `## Ответы и разборы` в конце файла.

Тело файла, по вопросу на блок:

```markdown
## Q1 [topic: <slug>] [difficulty: easy|medium|hard]
**Сценарий**: <production-ситуация, 2-4 предложения>
- A) <вариант>
- B) <вариант>
- C) <вариант>
- D) <вариант>
```

В конце файла, после горизонтальной черты:

```markdown
## Ответы и разборы

### Q1
**Ответ**: B
**Разбор**: <почему B чинит root cause и чем плохи остальные, 1-3 предложения>
```

Требования: ровно 4 варианта; один верный; верные ответы распределять по буквам A-D
равномерно; минимум один дистрактор класса «слой поверх» (retry-обёртка, бамп лимита,
форсинг, костыль); topic-теги переиспользовать между вопросами модуля (по ним строятся
слабые темы).

---

## Token discipline

- Study/quiz/practice сессии: лёгкие, без сабагентов и без web. Web и deep research только
  в Authoring, и то точечно.
- Не перечитывать главу целиком в quiz-режиме: достаточно банка.
- Разборы короткие, не лекции. Пользователь всегда может задать follow-up.
