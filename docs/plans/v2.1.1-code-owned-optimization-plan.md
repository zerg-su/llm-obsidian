# v2.1.1 code-owned optimization plan

Status: proposed for Fable/high review. Do not implement until the plan review
is accepted.

Baseline: `v2.1.0` / `94cc6cfb70e577db8034a1dcf0009b3e0717b672`.
All work belongs on `task/v2.1.1-code-owned-optimizations`; v2.1.0 remains an
immutable fallback.

## Objective

Reduce repeated model context, model calls, acceptance latency, and manual
coordination by moving deterministic behavior into repo-owned code. Preserve or
increase correctness, review depth, safety, and observable user behavior.

The model must retain semantic work: interpreting a free-form request, choosing
truly relevant context from bounded candidates, implementing the task, judging
review findings, resolving scope/security questions, and writing the final
human summary. State transitions, discovery, validation, correlation,
selection, accounting, cleanup, and retry decisions should be code-owned when
their inputs and outputs can be expressed as a strict contract.

## Non-negotiable invariants

1. No optimization may silently omit context, a required review, a changed
   acceptance cell, a blocking finding, or a cleanup failure.
2. Unknown dependency or routing drift invalidates cached evidence fail-closed.
3. Telemetry is content-free, local, bounded, best-effort, and never blocks or
   changes a model response. Prompt text, assistant text, reasoning, tool input,
   tool output, commands, file contents, absolute paths, credentials, and raw
   API bodies are never persisted.
4. Provider telemetry remains opt-in and disabled safely when unsupported. A
   missing collector degrades observability only, never the task pipeline.
5. Existing v2.1.0 defaults and semantic contracts remain compatible. Fresh
   macOS installation is primary; Linux remains basic; Windows is unsupported.
6. Full release acceptance still requires a complete set of valid green cell
   evidence. Reuse changes how unchanged evidence is obtained, not which cells
   are required.

## Workstream 1 — provider-native, message-level telemetry

### Provider facts

- Claude Code has built-in OpenTelemetry metrics and log events. Current
  official documentation describes a `prompt.id` correlation attribute on
  prompt-related events and model, duration, input/output/cache-token counts,
  cost, optional request ID, and query source on
  `claude_code.api_request`. Prompt content and tool details are off by
  default. These attributes remain hypotheses until current-runtime capture
  confirms them; beta traces are not required for this plan.
- Codex has built-in OpenTelemetry logs. `codex.api_request` records request
  duration/status and `codex.sse_event` records the stream event kind plus token
  counts on `response.completed`; conversation/model/sandbox metadata is
  attached by the runtime. Project config cannot set Codex `otel`, so user-level
  configuration must be explicit and idempotent.

Official references:

- Claude monitoring: https://code.claude.com/docs/en/monitoring-usage
- Claude hooks: https://code.claude.com/docs/en/hooks
- Claude Agent SDK observability: https://code.claude.com/docs/en/agent-sdk/observability
- Codex configuration/OTel manual: https://learn.chatgpt.com/docs/config-file/config-advanced.md

The Codex URL above is emitted by the current official Codex manual helper but
must be re-fetched during implementation. If the page route is unavailable,
replace it with the then-current authoritative Codex configuration source
rather than preserving a dead citation.

### Design

1. Add a stdlib-only, loopback-only OTLP/HTTP JSON receiver. Configure both
   runtimes to use their JSON OTLP protocol so the repository does not need a
   protobuf runtime or an external collector.
2. Accept only bounded `/v1/logs` and `/v1/metrics` requests from
   `127.0.0.1`/`::1` carrying the per-install bearer token. Reject
   unauthenticated, oversized, malformed, compressed-unknown, or non-OTLP
   payloads. Do not implement a general forwarding proxy.
3. Normalize only a strict allowlist of Claude and Codex events into a new
   `.vault-meta/model-events.jsonl` schema:
   runtime, safe session/conversation identifier, hashed prompt/request
   correlation, model, event kind, query source enum, status, sequence, token
   counts, cache-token counts, cost where supplied, and duration. Drop all
   unknown string attributes rather than storing them.
4. Reuse the repository's `fcntl`, atomic-write, permissions, rotation, and
   sanitization patterns. Collector failure is fail-open for the model session
   and produces at most one content-free degraded notice per session.
   `.vault-meta/model-events.jsonl`, its `.1` rotation sibling, and its lock
   are explicitly gitignored, size-capped, and excluded from every Stop-hook
   scoped commit. A regression test must prove all three exclusions.
5. Add a dry-run-first `model-telemetry.py check|install|uninstall|status`
   adapter. `install --apply` may, with explicit user authority, create the
   macOS launchd service and atomically merge user-level Claude environment and
   user-level Codex OTel configuration. Preserve unrelated configuration and
   keep timestamped backups. Project `.codex/config.toml` must not pretend to
   enable OTel because Codex ignores that key at project scope. The Claude
   merge is deliberately user-scoped too; unrelated sessions can reach the
   machine collector but cannot be persisted without an exact vault binding.
6. Keep `OTEL_LOG_USER_PROMPTS`, Claude tool/content/raw-body flags, and Codex
   `log_user_prompt` disabled. Do not enable Claude enhanced beta tracing by
   default. Stable log events already cover per-model-request token and latency
   accounting.
7. Extend `session-preflight.py` with a fast non-blocking capability check and
   a once-per-session degraded notice when telemetry was requested but the
   collector/config/runtime capability is missing.
8. Extend `pipeline-stats.py` with bounded summaries by runtime, model,
   session/task, request source, and stage. Reports show counts, tokens, cache
   ratio, p50/p95 latency, and missing-telemetry coverage; they never print
   provider payloads or model text.
9. Correlate provider events to existing project/task/lane/operation metadata
   through safe session bindings. Never infer a task from prompt similarity.

### Machine topology, authentication, and vault scoping

1. Run exactly one collector per machine, following the existing MCP gateway
   pattern. Its launchd label, canonical loopback port, schema/protocol version,
   installed code location, and owner manifest live in
   `~/.config/llm-obsidian/model-telemetry/`, not in a checkout. The launchd
   job executes an atomically installed versioned copy under the user's local
   data directory, so deleting a worktree cannot break the service.
2. A second vault's `install --apply` reuses a healthy compatible collector.
   It never replaces the service, port, token, or user configuration silently.
   Incompatible ownership/schema drift is reported and installation stops
   before mutation.
3. Installation generates a random bearer token in an owner-only `0600`
   secrets file outside Git and configures both provider exporters to send it
   through their supported OTLP headers. The receiver uses constant-time token
   comparison and accepts no unauthenticated request, even from loopback.
4. Each SessionStart/preflight registers the exact provider session with a
   canonical vault/project/task binding through a bounded authenticated local
   control call. Linked worktrees resolve to the same project identity and
   dispatched tasks route telemetry to their recorded origin vault.
5. Events that arrive before a binding may be held only in a small in-memory
   TTL buffer. Unbound events from unrelated Claude/Codex sessions are dropped
   without persisting their identifiers or payload attributes; only anonymous
   aggregate drop counters may enter machine-level health status. No vault may
   receive another vault's events.
6. On Linux, `check` and `status` report capabilities without mutation.
   `install --apply` refuses with a documented foreground/manual command until
   a tested service manager exists; it must not leave partial provider config.
   Windows remains unsupported.

### Stage telemetry

Add explicit monotonic stage spans for dispatch, reviewer launch/callback,
verify, reap, acceptance sandbox setup, model wait, child settling, proof, and
cleanup. Join these lifecycle spans with provider request events through the
exact session/task binding. This must answer whether a five-minute acceptance
cell spent time in setup, model inference, review/reap, or surface shutdown.

## Workstream 2 — content-addressed acceptance reuse

1. Add a declarative dependency manifest for each acceptance cell/scenario and
   shared runtime group. Use a stdlib-readable format (JSON or TOML); do not add
   a YAML parser solely for this file.
2. Compute a `cell_fingerprint` from exact content hashes of the skill,
   fixture, scenario, declared transitive scripts/config/schemas, resolved
   model/effort, Claude/Codex and cmux versions, OS/architecture, acceptance
   phase, and runner contract version.
   Include a provider model snapshot/version when the runtime exposes one. If a
   hosted alias has no immutable snapshot identifier, the bounded freshness
   rule remains mandatory.
3. Store the dependency expansion and safe reason codes alongside each result
   so `--explain-selection` can state why a cell was reused or invalidated.
4. Reuse only prior `pass`/`n-a` evidence with an identical cell fingerprint,
   compatible environment, and a default seven-day freshness window. The
   window may be shortened freely; increasing it beyond 14 days requires an
   explicit reviewed contract change. Never reuse
   `fail`/`blocked`, partial, unclean, manually edited, or unknown-provenance
   results.
5. Any changed repository path not covered by the dependency manifest must
   conservatively invalidate all live cells until classified. Changes to
   shared launch, routing, security, callback, cleanup, or evidence validation
   code invalidate every affected runtime/scenario group.
6. Preserve `--restart` as an explicit full rerun. A final release report is
   complete only when every currently installed skill/runtime cell has current
   valid evidence, whether freshly executed or safely reused.
7. Make checkpoint writes atomic after every cell and preserve first-failure
   evidence. Do not cache model prose as truth; cache the validated typed
   verdict plus code-owned proof and provenance.

## Workstream 3 — runner-first skill compaction

1. Measure current loaded bytes/estimated tokens for every orchestration skill
   and its normal happy path before editing.
2. Keep each top-level `SKILL.md` focused on trigger, semantic inputs, safety
   boundaries, the exact runner command, typed result, and escalation rules.
3. Move legacy/classic mechanics, operator diagnostics, schemas, and long
   examples into routed references. The normal v3 path must not read them;
   legacy/failure paths must load the exact reference they need.
4. Remove instructions that ask the model to reproduce code-owned shell
   mechanics. Do not remove substantive judgment or security guidance.
5. Add a runtime body-budget check in addition to the existing registry
   description budget. Track bytes, lines, and a deterministic token estimate
   per skill and for the dispatch/review/reap chain. Require an explicit
   baseline update when a budget increases.
6. Add contract lint tests proving the compact entry files still expose every
   required command, fail-closed boundary, callback rule, same-session verify
   rule, process-exit close rule, and compatibility route.
7. Compare live behavior before/after on representative Claude and Codex
   dispatch-review-reap cells. Token reduction is accepted only when typed
   outputs, product diff, review outcome, cleanup proof, and user-visible
   behavior remain equivalent.
8. Skill compaction necessarily changes the relevant skill fingerprints.
   Workstream 2 must therefore invalidate and rerun those cells rather than
   treating the smaller instructions as documentation-only changes.

## Implementation order

1. Capture immutable v2.1.0 functional and size/latency baselines.
2. Before freezing the schema, capture bounded one-message OTLP/HTTP JSON from
   the currently installed Claude and Codex runtimes into an owner-only
   temporary directory, inspect it without committing content, and reconcile
   every provider-fact claim above. Golden fixtures are minimal synthetic
   content-free derivatives of those captures, not copies of prompts or
   responses. When a prompt/request correlation ID is absent, use the exact
   provider session/conversation ID plus monotonic event sequence and timestamp;
   never invent a relationship from text. Re-fetch every cited provider source
   through the official documentation route and empirically confirm that Codex
   ignores project-scoped `otel` before committing to a user-only config
   mutation. Then implement the strict schema, parsers, receiver, fixtures, and
   reports before wiring user configuration.
3. Add dry-run/check/install adapters and real one-message Claude and Codex
   telemetry dogfood using inexpensive explicit test models. Keep production
   defaults unchanged.
4. Implement per-cell fingerprints, conservative dependency expansion,
   selection explanations, and cross-commit reuse.
5. Compact runner-first skills and use the new telemetry to measure actual
   context/token changes.
6. Run focused unit/integration suites after each workstream, then all of
   `make test`, affected live acceptance cells, and the complete final evidence
   gate required by the new fingerprint contract.
7. Run a final full Fable/high implementation review. Resolve findings in the
   same reviewer lane. Do not merge, tag, push, publish, or release without a
   separate user decision.

## Required tests

- Golden OTLP/HTTP JSON fixtures for current Claude and Codex events.
- Persistence-exclusion tests for prompt/assistant/tool content and rejection
  tests for raw-body event types, unknown event strings, absolute paths,
  secrets, oversized/decompression-bomb payloads, malformed numbers, unknown
  compression, concurrency, permissions, rotation, and collector restart.
- Installer dry-run/idempotency/backup/uninstall tests with existing user
  config tables and denied writes; no real home mutation in hermetic tests.
- Multi-vault ownership tests: compatible second install reuses the service,
  incompatible drift refuses before mutation, bound events reach only their
  exact vault, unbound events persist no identifiers, and bearer failures are
  rejected.
- Control-endpoint tests matching the OTLP boundary: authentication,
  size/schema/method limits, replay/idempotency, wrong-vault and stale-session
  bindings, path confinement, concurrency, and restart behavior.
- Telemetry-unavailable and collector-crash tests proving model/task behavior
  and exit status are unchanged.
- Fingerprint tests for direct, transitive, shared, model/config/toolchain,
  freshness, unknown-path, failed-result, partial-report, and tamper cases.
- Skill body-budget and runner-contract lint tests.
- Representative live Claude/Sonnet and Codex/Terra telemetry cells, followed
  by production-default routing checks. Test overrides must not alter product
  defaults.
- Before/after quality evidence for dispatch-review-reap: identical fixture
  bytes, exact one commit, typed review approval, archive/reap proof, clean
  disposable clone, exact surface close, and no fallback cleanup.

## Acceptance criteria

1. No observed semantic or safety regression in unit, integration, live, and
   Fable review evidence.
2. Normal dispatch/review/reap skill context is materially smaller, with a
   committed body budget preventing regression.
3. A code change reruns exactly the cells whose fingerprints changed; an
   unknown dependency reruns conservatively rather than reusing evidence.
4. Every supported Claude/Codex model request can produce a content-free local
   event with token and latency counters when telemetry is enabled.
5. Telemetry-disabled, unsupported, or failed states do not change model
   answers, task lifecycle, commits, review decisions, or cleanup.
6. A machine-readable report identifies the dominant latency stage for each
   live cell and quantifies tokens saved by skill compaction.

## Rollback

Disable/uninstall the telemetry adapter, ignore the new evidence cache, and
restore the pre-compaction skill entry files. Uninstall removes the launchd
service and restores or reverses only the managed Claude/Codex configuration
keys using the recorded install manifest and timestamped backups. The immutable
`v2.1.0` tag remains the known-good fallback. No v2.1.1 data migration may
make v2.1.0 unable to read the vault or operate without telemetry.
