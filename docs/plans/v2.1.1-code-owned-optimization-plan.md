# v2.1.1 code-owned optimization plan

Status: committed review candidate after user clarification. Product
implementation starts only after Fable/high verification binds the exact Git
HEAD in validated review history; the SHA is recorded there rather than in this
self-referential file.

Baseline: `v2.1.0` / `94cc6cfb70e577db8034a1dcf0009b3e0717b672`.
All work belongs on `task/v2.1.1-code-owned-optimizations`; v2.1.0 remains an
immutable fallback.

## Objective and invariants

Reduce repeated model context, model calls, acceptance latency, and manual
coordination by moving deterministic behavior into repo-owned code without
reducing semantic quality, review depth, safety, or observable correctness.

- The model retains interpretation, semantic context selection, implementation
  judgment, finding classification, and human summaries. Deterministic
  discovery, validation, state transitions, accounting, and cleanup are
  code-owned.
- Unknown dependencies or routing drift invalidate cached evidence fail-closed.
- Telemetry is content-free, local, bounded, best-effort, and cannot affect a
  model answer or task result.
- No OpenTelemetry collector, launchd service, provider config mutation, new
  event journal, or cmux timing indicator is in scope.
- Existing v2.1.0 defaults and compatibility paths remain valid. Fresh macOS is
  primary; Linux remains basic; Windows is unsupported.

## Workstream 1 — simple turn and stage telemetry

1. Reuse `.vault-meta/pipeline-events.jsonl`, its existing fixed schema,
   `fcntl` writer, one-megabyte rotation, and `pipeline-stats.py`. Add no second
   telemetry store.
2. In the shared Claude/Codex runtime adapter, on `UserPromptSubmit` atomically
   record a session-scoped start marker. On entry to `Stop`, remove it and emit
   `model-turn` with `duration_ms` before Stop-hook reindex/commit work starts.
   The duration covers reasoning, tool calls, and the final answer, but not Stop
   processing.
3. Resolve the exact origin vault for coordinator, dispatch, and reviewer
   sessions from trusted session/task/review metadata. Internal agents without
   their own interactive session are part of the parent turn. Missing session
   ID is a silent no-op; never guess from PID, cwd, prompt, or timestamps.
4. If the exact session starts or submits a new prompt with a stale marker,
   emit `model-turn-incomplete` without a duration and replace/remove the
   marker. Incomplete events never enter latency percentiles. `Stop` with no
   marker for that exact session is a silent no-op.
5. Emit `duration_ms` for existing exact runner boundaries covering dispatch,
   review, verify, reap, acceptance setup/model wait/proof/cleanup, and surface
   shutdown. Do not synthesize spans for unavailable boundaries.
6. Extend `pipeline-stats.py` with turn counts, incomplete counts, and p50/p95
   duration by runtime/session role and lifecycle stage. Store no prompts,
   responses, reasoning, tool data, commands, absolute paths, or error text.

## Workstream 2 — content-addressed acceptance reuse

1. Add hand-maintained `config/acceptance-cells.toml` for cell/scenario and
   shared runtime-group dependencies. Use only stdlib `tomllib`; no YAML parser
   or new dependency.
2. Compute a `cell_fingerprint` from sorted exact content hashes of the skill,
   fixture, scenario, declared transitive scripts/config/schemas, runtime and
   cmux versions, OS/architecture, acceptance phase, runner contract version,
   and canonical production model generation. Unknown changed paths invalidate
   all live cells until classified. The changed-path set is relative to each
   evidence row's provenance commit and includes staged, unstaged, and
   non-ignored untracked paths.
3. Derive primary generations automatically from effective production routing,
   with explicit include/exclude overrides in the acceptance TOML. Temporary
   live-test overrides never become primary generations. Canonicalize
   `gpt-5.6-sol` and `gpt-5.6-terra` as `codex:5.6`; Claude uses its registered
   large-release name such as `opus` or `fable`. Reasoning effort does not
   affect the fingerprint. A Sonnet/Terra acceptance override is likewise
   excluded from fingerprint generation: the row carries the effective
   production generation while its actual executed model remains explicit in
   provenance.
4. Save the content-free generation snapshot under `.vault-meta/acceptance/`.
   A changed generation causes one SessionStart notice and marks dependent live
   cells stale; it never starts tests in the background. There is no TTL. This
   deliberately accepts silent provider drift behind an unchanged hosted alias;
   every reused row must expose its evidence age in schema 2 and in
   `--explain-selection` so the final gate makes that risk visible.
5. Upgrade `.vault-meta/acceptance/latest-live.json` to schema 2 with each
   row's fingerprint, expanded dependencies, generation, provenance, and
   bounded reuse/invalidation reason, evidence age, and a canonical
   `row_integrity_sha256` over the typed result, fingerprint, and provenance.
   The loader recomputes the integrity hash and rejects accidental/manual
   edits. This existing report is the only cache; reuse is local to its
   worktree across commits, not shared across worktrees.
6. Reuse only validated `pass`/`n-a` rows with an identical fingerprint and
   provenance. Never reuse `fail`, `blocked`, partial, tampered, manually
   edited, or unknown-dependency evidence. Schema 1 evidence is not eligible
   for cross-commit reuse. `--restart` always executes the full matrix.
7. Checkpoint atomically after every cell and preserve first-failure evidence.
   A final release report still requires valid current evidence for every row;
   reuse changes execution selection, not matrix completeness.

## Workstream 3 — all-skills runner-first optimization

1. Measure description and body bytes, lines, and a deterministic token
   estimate for every installed skill and normal-path orchestration closure.
   Extend the existing budget checker with a committed generated JSON body
   baseline; increases require an explicit baseline update.
2. Apply one common optimization pass across the complete skill inventory, but
   edit a skill only when normal-path context shrinks, repeated mechanics move
   into shared repo-owned code or routed references, semantic/safety behavior
   remains intact, and contract plus live-equivalence tests pass. Do not add
   bespoke runners merely to claim smaller prompts.
3. Keep top-level `SKILL.md` files focused on triggers, semantic inputs, safety
   boundaries, canonical runner commands, typed outputs, and escalation rules.
   Move classic/interim/legacy/recovery mechanics and long diagnostics to
   references loaded only on the matching path.
4. Preserve `dispatch-resolver.py` and `reap-runner.py` as canonical code; do
   not create parallel implementations. Compact dispatch/review/reap/close
   normal paths around their existing runners while retaining compatibility.
5. Keep Phase 1 ranking behavior: code resolves one exact repo and approved
   plan, returns at most five ranked context candidates, and the model chooses
   semantic relevance. Add a bounded retrieval timeout and negative tests for
   missing/ambiguous plan, explicit cross-session plan, malformed retrieval,
   timeout, and lexical degradation. Do not change retrieval ranking in this
   release.
6. Contract lint must retain required commands, fail-closed boundaries,
   callbacks, same-session review verification, process-exit surface closing,
   and every compatibility route. A skill with no safe common optimization may
   remain unchanged without blocking v2.1.1.

## Implementation and verification order

1. Capture v2.1.0 skill-size, behavior, and latency baselines.
2. Obtain Fable/high approval of this revised plan and resolve plan findings in
   the same reviewer lane before product implementation.
3. Implement telemetry, acceptance fingerprints, then skill optimization,
   running focused unit/integration tests after each workstream.
4. Telemetry tests cover concurrent session markers, stale/incomplete turns,
   Stop without a marker, missing IDs, exact task/reviewer origin routing,
   privacy rejection, rotation, failure no-op behavior, percentiles, stage
   accounting, and an explicit Codex `UserPromptSubmit`-to-`Stop` fixture
   through the shared adapter.
5. Acceptance tests cover direct/transitive/shared/unknown dependencies,
   configuration/toolchain/platform drift, generation canonicalization,
   ignored effort and the exact Sonnet-production-generation override case,
   evidence age, row-integrity tampering, staged/unstaged/non-ignored-untracked
   dirty paths, schema-1 incompatibility, cross-commit same-worktree reuse,
   failures, checkpoint recovery, selection explanations, and `--restart`.
6. Skill tests cover body budgets, runner/reference routing, safety contracts,
   Phase 1 timeout/failures, and before/after deterministic normal-path size and
   token-estimate evidence.
7. Run all of `make test`. Then run only changed fingerprint live cells using
   Claude Sonnet and Codex Terra. A shared runner/routing/security/callback/
   cleanup change invalidates every dependent cell. Separately prove that
   production routing still selects Opus/Fable and Sol as configured.
8. Compare v2.1.1 with v2.1.0: typed outputs, task product, review verdict,
   commit/proof, cleanup, and user-visible behavior must be equivalent or
   better. Token/context savings do not excuse a quality regression.
9. After green tests and affected live cells, stop before the final Fable/high
   implementation review and ask the user whether v2.1.1 needs additions.
   Only after that decision run the final review, resolving findings in the
   same reviewer lane.
10. Do not merge, tag, push, publish, or release without a separate user
    decision.

## Rollback

`v2.1.0` remains the known-good fallback. v2.1.1 adds no external service or
provider configuration to remove. Rollback restores the previous skill entry
files; the first v2.1.0 acceptance run must use `--restart` (or delete the
derived schema-2 `.vault-meta/acceptance/latest-live.json`) because v2.1.0
rejects that report version. Runtime telemetry files remain derived, gitignored
state, and vault content remains readable by v2.1.0.
