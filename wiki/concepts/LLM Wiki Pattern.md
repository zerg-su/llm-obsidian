# LLM Wiki Pattern

Паттерн построения персистентной компаундящей базы знаний с помощью LLM. Сформулирован [[Andrej Karpathy]]. Ключевой инсайт: вместо того чтобы re-derive знания из сырых документов на каждый запрос (RAG), LLM инкрементально строит и поддерживает структурированную вики, которая обогащается с каждым добавленным источником.

---

## Базовая идея

Большинство AI-knowledge инструментов работают как RAG: индексировать сырые документы, доставать чанки на момент запроса, генерировать ответ. Ничего не накапливается. Спросите вопрос, для которого нужно пять документов — LLM пересобирает фрагменты каждый раз.

Wiki-паттерн другой. Когда приходит новый источник, LLM читает его, извлекает важное, интегрирует в вики: обновляет entity-страницы, отмечает противоречия, обогащает синтез. Cross-references уже на месте. Знания компилируются один раз и держатся актуальными.

**Вики это персистентный, компаундящий артефакт.** Человек курирует источники и задаёт вопросы. LLM пишет и поддерживает всё остальное.

---

## Три слоя

```
.raw/       Layer 1 — immutable source-документы
wiki/       Layer 2 — сгенерированная LLM база знаний
CLAUDE.md   Layer 3 — схема и инструкции для LLM как её поддерживать
```

LLM владеет Layer 2 целиком. Создаёт страницы, обновляет их при поступлении новых источников, поддерживает cross-references, держит всё консистентным. Человек читает; LLM пишет.

---

## Операции

**Ingest** — положить источник в `.raw/`, сказать LLM обработать. LLM читает источник, обсуждает ключевые тейкэвэи, пишет summary-страницу, обновляет entity и concept-страницы, логирует операцию. Один источник обычно затрагивает 8-15 wiki-страниц.

**Query** — задать вопрос. LLM читает index чтобы найти релевантные страницы, синтезирует ответ с цитатами. Хорошие ответы возвращаются в вики.

**Lint** — периодический health-check. Найти orphan-страницы, dead links, stale claims, missing cross-references.

---

## Index и Log

**index.md** — content-oriented. Каталог всех страниц с одностроковыми summary, организованный по категориям. LLM читает первым на каждый запрос чтобы найти релевантные страницы.

**log.md** — хронологический. Append-only запись каждого ингеста, запроса, lint-pass'а. Parseable: `grep "^## \[" log.md | head -10`

---

## Почему это работает

Скучная часть поддержания knowledge-base это bookkeeping: обновление cross-references, отметки когда новые данные противоречат старым утверждениям, актуализация summary. Люди забрасывают вики потому что нагрузка обслуживания растёт быстрее ценности. LLM не скучают. Вики остаётся поддерживаемой потому что стоимость поддержки близка к нулю.

На малом масштабе (~100 источников, ~сотни страниц) index-файла достаточно. Никакой vector-database, никаких embeddings, никакой инфраструктуры. Просто markdown-файлы.

---

## Сравнение с RAG

| Dimension | LLM Wiki | Semantic RAG |
|---|---|---|
| Поиск | Читает index, следует по ссылкам | Similarity search над embeddings |
| Инфраструктура | Просто markdown-файлы | Embedding-модель + vector DB |
| Стоимость | Только токены | Ongoing compute + storage |
| Поддержка | Прогон lint | Re-embed при каждом изменении |
| Лимит масштаба | Сотни страниц | Миллионы документов |
